Methodology

How this was calculated.

Packing (executors per node)

Executors per node = min(floor((node_cores − reserve_cores) / executor_cores), floor((node_memory_gb − reserve_memory) / executor_memory_gb)). Efficiency score = 100 × (1 − waste), where waste = max(CPU waste, memory waste) after packing. Implementation: icea/packing.py.

Cost (daily and monthly)

Hourly cluster cost = node_hourly_price × node_count. Daily cost = hourly_cluster_cost × (avg_runtime_minutes / 60) × jobs_per_day × utilization_factor. Monthly cost ≈ daily_cost × 30. Implementation: icea/cost_model.py.

Waste and waste cost

Waste is the unused fraction of allocated capacity (CPU or memory left over after packing, or capacity not fully utilized by the workload). Waste cost = daily_cost × waste × 30.

Understanding This Report

Brief explanations of the metrics and terms used in this report.

Efficiency score

A 0–100 score showing how well CPU and memory are used on each node. It is based on the fraction of node capacity used by executors; the remainder counts as waste. Higher is better. Low scores usually mean executor size does not fit the node well.

Waste and waste cost

Waste is the share of node CPU and memory reserved for executors but not fully used by your workload, plus any capacity left over because executors do not pack evenly. Waste cost is the estimated dollar value of that unused capacity at your node price and usage.

Executors per node

The number of executors that fit on one worker node given your executor cores, memory, and the node’s capacity. More executors per node usually improve parallelism but can increase scheduling and GC overhead if the number is too high.

Resource utilization (CPU and memory)

The percentage of available node CPU and memory allocated to executors. The rest is reserved for the OS and system processes. Utilization is limited by the tighter of the two (CPU or memory).

Recommended configuration

An alternative executor size (cores and memory) that improves packing and reduces waste for the same node type. Savings are estimates; actual results depend on workload behavior and tuning.

Definitions

Key terms used in this report.

TermDefinition
NodeA worker machine in the cluster (e.g. an EC2 or Dataproc instance). Each node has a fixed number of vCPUs and memory (GB) available for executors after reserves.
ExecutorA Spark executor process running on a node. Each executor is configured with a fixed number of cores and memory (GB). Multiple executors can run on a single node.
PackingThe process of fitting executors onto nodes. Executors per node = min(floor(available_cores / executor_cores), floor(available_memory / executor_memory)), after reserving capacity for the OS. The limiting resource (CPU or memory) determines the result.
WasteUnused capacity: CPU or memory left over after packing, or capacity allocated to executors but not fully utilized by the workload. Reported as a fraction (0–1) and as an estimated monthly cost (waste cost).
Efficiency scoreA 0–100 score derived from the waste fraction: 100 × (1 − waste). Higher means better use of node capacity; low scores indicate poor fit between executor size and node.
Reserve (cores and memory)Capacity reserved on each node for the OS, daemons, and system processes. This capacity is not used for executors and is excluded from packing calculations.
Utilization factorAn optional multiplier (0–1) for shared clusters, representing the fraction of time the cluster is used for this workload. Applied to scale daily cost when the cluster runs other jobs.
